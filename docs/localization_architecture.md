# Архитектура и регламент локализации Antigravity Features

Этот документ описывает устройство системы локализации (RU/EN) для Chromium-страниц в IDE Antigravity, типичные проблемы, способы их предотвращения и правила проверки.

---

## 1. Как устроена локализация

Локализация состоит из двух частей:
1.  **Бэкенд Node.js (`index.js` в `antigravity-features`):**
    *   Считывает сохранённый язык пользователя из файла `antigravity_lang.txt` на жестком диске.
    *   При обнаружении новых вкладок Chromium через CDP инжектирует в них скрипт локализации, пробрасывая в контекст переменную `window.__antigravity_saved_lang`.
    *   Раз в секунду проверяет, не записал ли Chromium-инжектор новое значение в `window.__antigravity_pending_lang_change` (при клике пользователем на RU/EN). Если записал — сохраняет его в `antigravity_lang.txt`.
2.  ** Chromium-инжектор (`localization_injected.js`):**
    *   Считывает язык из `localStorage.getItem('antigravity_lang')` или `window.__antigravity_saved_lang`.
    *   **Динамический перевод:** Использует `MutationObserver` для отслеживания изменений в DOM. Когда React рендерит новые элементы (вкладки, чат, настройки, поповеры), обсервер ловит мутации и вызывает `translateTree()` для перевода текстовых узлов.
    *   **Синхронизация:** Каждые 500мс проверяет значение `window.__antigravity_saved_lang` и при его обновлении на лету переключает язык в `localStorage` и вызывает `updateUI()`, переводя страницу без её физической перезагрузки.

## 2. Чего КАТЕГОРИЧЕСКИ нужно избегать в коде

1.  **Бесконечных циклов `window.location.reload()`:**
    *   Никогда не используйте автоматический бэкап и перезапись *всего* `localStorage` с перезагрузкой страницы. IDE хранит там временные служебные токены, которые постоянно меняются. Перезапись этих токенов вызывает циклическую перезагрузку страницы, утечку памяти и падение рендерера Electron в серое окно ("White Screen of Death").
    *   Синхронизация должна затрагивать *только* ключ `antigravity_lang` и происходить в памяти в реальном времени.
2.  **Блокировок перевода флагами в обсервере:**
    *   Не выставляйте флаг `isTranslating = true` внутри самого `MutationObserver`. Функции `translateTree` и `translateTextNode` проверяют этот флаг и сразу выходят без перевода, если он равен `true`.
    *   Для предотвращения рекурсии при записи перевода в DOM используйте пару `stopObserving()` перед переводом и `startObserving()` после него. Это на 100% безопасно, так как операции перевода синхронны, а JS однопоточен.
3.  **Слепого игнорирования контейнеров в `shouldIgnore`:**
    *   Не используйте восходящий обход родителей (`parent.parentElement`) для блокировки перевода, если вы предварительно не проверили Белый список классов (White List). Поскольку настройки, меню и поповеры вложены в основной контейнер приложения React (который содержит классы чата), простой обход родителей заблокирует перевод всей панели настроек.
    *   Интерактивные элементы (кнопки, вкладки, настройки, меню, списки) должны выходить по белому списку классов (`settings`, `modal`, `dialog`, `btn`, `dropdown` и т.д.) до того, как обход наткнётся на родительский контейнер чата.

## 3. Регламент безопасного изменения кода

Перед отправкой коммита или завершением задачи агент обязан:
1.  **Проверить синтаксис всех затронутых скриптов:**
    ```bash
    node --check index.js
    node --check localization_injected.js
    ```
2.  **Запустить деплой в рантайм:**
    ```bash
    node mcp-suite/deploy.js
    ```
3.  **Провести автоматизированную валидацию через CDP:**
    Запустить скрипт проверки локализации (например, `verify_localization.js`), который подключится к Chromium, сымитирует переключение языков, динамический ввод текста и проверит перевод плейсхолдеров и окон настроек в DOM.
4.  **Привести логи валидации:**
    Все логи успешного перевода должны быть приложены в ответе пользователю в качестве Proof of Work.
